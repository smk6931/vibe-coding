## 1. member 모델 변경 (사용자 직접 입력 ID)

### 1-1. 정책

- 회원가입 시 사용자가 직접 ID 정함 (예: `smk6931`)
- ID = 영문 소문자 + 숫자 + 언더스코어 (3~20자)
- ID 변경 불가 (또는 6개월 1회 강한 제한 — 추후 결정)
- 닉네임 별도 컬럼 안 둠 (ID 가 곧 표시 이름)
- 단 한국어 표시 원하면 `display_name` (변경 자유, unique 아님) 옵션 추가

### 1-2. 모델

```python
class Member(...):
    id            = models.SlugField(primary_key=True, max_length=20)  # 'smk6931'
    display_name  = models.CharField(max_length=50, blank=True)         # '손명관' (한국어 표시명, optional)
    email         = models.EmailField(unique=True)                      # 가입 시 이메일 인증 XX
    password      = models.CharField(max_length=128)                    # Django set_password
    name          = models.CharField(max_length=50)                     # 실명 (입금자 확인용)
    age           = models.PositiveSmallIntegerField(null=True, blank=True)
    phone         = models.CharField(max_length=20)                     # 전화번호
    role          = models.CharField(max_length=10, default='member')   # 'member' / 'admin'
    provider      = models.CharField(max_length=10, default='email')    # 'email' / 'kakao' / 'google'
    avatar_url    = models.URLField(blank=True)
    bio           = models.TextField(blank=True)
    is_active     = models.BooleanField(default=True)                   # 휴면
    created_at, updated_at
```

### 1-3. URL 노출

- 사용자 프로필: `/u/smk6931` (id 직접)
- API: `/api/v1/member/smk6931`
- 표시: 기본 = `id`, `display_name` 있으면 그것 우선

### 1-4. 가입 폼 (Sprint 2)

```
- ID            * (영문/숫자/_, 3~20자, 영구 불변)
- 비밀번호      *
- 비밀번호 확인 *
- 이메일        *
- 이름 (실명)   * (입금자 확인용)
- 나이          (선택)
- 전화번호      *
- (선택) display_name (한국어 표시명)
```

너무 깊은 정보 안 받음. 입금자 확인 + 신청 안내가 본질.

## 2. PK / FK 표준 전환 (사용자 헷갈림 해소 + 면접 안전)

### 2-1. 표준 채택

**모든 테이블 = integer PK auto-increment (Django 기본)**.
**Domain entity 는 별도 `slug` 또는 `code` unique 컬럼**:

| 종류 | PK | 외부 노출 키 | URL |
|------|-----|----------|------|
| Domain (event, post, inquiry, site_config) | integer auto | **slug** (unique) | `/events/evt-week1-20260510-1400` |
| Member (특수) | **id (사용자 입력 string)** | 동일 | `/u/smk6931` |
| Transaction (event_registration, event_testimonial) | integer auto | **token** (unique nanoid 12자, 외부 노출 시) | `/registrations/{token}` |

### 2-2. join 은 integer FK

```python
class EventRegistration(models.Model):
    event   = models.ForeignKey(Event, on_delete=models.CASCADE)   # event_id (int FK)
    member  = models.ForeignKey(Member, on_delete=models.SET_NULL, null=True)  # member_id (string FK, member 만 예외)
```

### 2-3. URL 라우팅 패턴

```python
# views.py
class EventViewSet(viewsets.ModelViewSet):
    lookup_field = 'slug'   # URL 의 :id 부분이 slug 로 매칭

# urls.py
router.register('event', EventViewSet)
# → GET /api/v1/event/evt-week1-20260510-1400
```

내부에서는 integer FK 로 join, 외부 노출은 slug 로. **2단 분리**.

### 2-4. 변경된 ERD 미리보기

```
[event]
  ├─ id (PK, int auto)              ← 1, 2, 3, ...
  ├─ slug (unique, max 50)          ← 'evt-week1-20260510-1400'
  ├─ curriculum_slug (string, nullable)  ← 'oneday-week-1' (jsx 매칭)
  ├─ ...

[event_registration]
  ├─ id (PK, int auto)
  ├─ token (unique, nanoid 12)      ← 외부 노출 시 사용
  ├─ event_id (FK → event.id, int)
  ├─ member_id (FK → member.id, string, SET NULL on delete)
  ├─ ...
```

**join = integer FK (효율 ↑) + 외부 노출 = slug/token (가독성 ↑)**. 양쪽 다 안전.

## 3. EventApplication 폐기 → EventRegistration 으로 통합

### 3-1. 흐름

```
[Before — 3차]
비회원 → POST /event/:id/application → 운영자 응대 → POST /application/:id/convert → Member + Registration 생성

[After — 4차]
누구든 → /signup (간단 폼) → 회원 → POST /event/:slug/registration → status='pending' → 운영자 입금 확인 → status='confirmed'
```

### 3-2. 단순화 효과

- DB 테이블 1개 줄어듦 (8개 → 7개)
- "신청자 = 회원" 단순 룰
- 회원 자산 누적 (다음 회차 알림 가능)
- 진심 필터 (가입 의지 있는 사람만 신청)
- 단점: 첫 진입 마찰 ↑ (가입 + 신청 두 단계). 단 가입 폼 30초면 충분.

### 3-3. 5/10 회차 흐름

- **5/10 까지 = 카톡 신청 그대로** (백엔드 없음)
- **5/13 후 = 사이트 가입 + 신청 흐름**
- 5/10 회차 신청자 1~3명 → 5/13 후 운영자가 admin UI 로 회원 + 신청 데이터 직접 INSERT (seed)

## 4. 삭제 정책 — SET NULL + author_name 보존 + hard delete 옵션

### 4-1. 기본: 익명화 (SET NULL)

```python
class EventRegistration(models.Model):
    member       = models.ForeignKey(Member, on_delete=models.SET_NULL, null=True)
    author_name  = models.CharField(max_length=50)   # ← member 삭제 후에도 보존
    author_phone = models.CharField(max_length=20, blank=True)  # ← 입금자 확인용 보존

class EventTestimonial(models.Model):
    member       = models.ForeignKey(Member, on_delete=models.SET_NULL, null=True)
    author_name  = models.CharField(max_length=50)
    author_role  = models.CharField(max_length=50, blank=True)  # '비전공자', '대학생' 등
    content      = models.TextField()
```

회원 탈퇴 시:
- `member` row 는 그대로 (단 `is_active=False`)
- 또는 `member.id` row 자체 삭제 → SET NULL → registration·testimonial 의 `member_id = NULL`, `author_name` 보존

### 4-2. archive 테이블 안 만드는 이유

- 두 배 테이블 = 쿼리 복잡 (UNION 필요)
- author_name 보존 패턴이 동일 효과 + 단순
- 운영 레벨에서 진짜 archive 필요하면 그때 도입 (Sprint 5+)

### 4-3. hard delete 옵션 (개인정보 파기)

회원이 "내 모든 데이터 완전 삭제 (개인정보 보호법)" 요청:
```
DELETE /api/v1/me?purge=true
```
- member row hard delete
- registration·testimonial 의 `author_name` → "[삭제된 회원]"
- 통계는 유지 (member_id NULL)

## 5. 페이지네이션 + 검색 + Storage

### 5-1. 백엔드 페이지네이션 (표준 유지)

DRF `PageNumberPagination`:
```json
{
  "count": 42,
  "next": "/api/v1/post?page=3",
  "previous": "/api/v1/post?page=1",
  "results": [...]
}
```

### 5-2. 프런트 = 무한 스크롤 + 검색

| UI | 적합 패턴 |
|----|---------|
| event 목록 (회차 N건, 100 미만) | 단순 목록 (페이지네이션도 무한 스크롤도 불필요) |
| post 목록 (블로그) | **무한 스크롤** (Intersection Observer) |
| testimonial (후기) | **무한 스크롤** |
| event 검색 / post 검색 | `?q=keyword` + 결과는 무한 스크롤 |
| admin 목록 (registration 등) | **테이블 페이지네이션** (1/2/3/4 클릭) |

같은 백엔드 위에서 프런트 UX 만 다름. `useInfiniteScroll` 훅 1개 만들어 재사용.

### 5-3. 검색 — 백엔드 `?q=keyword`

```
GET /api/v1/post?q=supabase
GET /api/v1/event?q=강남
```
- Postgres `ILIKE` 또는 `to_tsvector` (full-text search)
- title + summary + tags 매칭
- 단순 LIKE 부터 시작, 누적되면 GIN 인덱스 + tsvector

### 5-4. Storage — **정적 / 동적 분리**

| 종류 | 위치 | 예시 |
|------|------|------|
| **빌드 타임 정적** (운영자 사진, 이벤트 썸네일, 강의 캡처) | `front/public/images/` (현재 그대로) | `/images/operator-photo.png` |
| **런타임 업로드** (모임 사진, 후기 사진, 회원 아바타) | **Cloudflare R2** (Sprint 3 도입) | `https://r2.vibe.me.kr/events/evt-w1-.../photo.jpg` |

**판정 기준**:
- 운영자가 git commit 으로 추가하는 사진 → `front/public/`
- 회원·운영자가 사이트 UI 로 업로드 → R2

**시점**:
- Sprint 1~2 = front/public/ 그대로 (운영자 사진, 1주차 캡처 등)
- **Sprint 3** = R2 셋업 (5/10 후 모임 사진 발생 시)

## 7. API 갱신 (Application 빠짐, Registration 통합)

### Auth (`/auth`)
```
POST   /signup              회원가입 (id, password, email, name, phone, age?, display_name?)
POST   /login               id + password → JWT
POST   /refresh             access 갱신
POST   /logout              refresh 폐기
GET    /me                  내 정보
PATCH  /me                  내 정보 수정 (id 변경 X)
DELETE /me?purge=true       hard delete (개인정보 파기)
```

### Event (`/event`)
```
GET    /event                       회차 목록 (?source=&type=&q=&page=)
GET    /event/:slug                 회차 상세
GET    /event/upcoming              가까운 미래 회차
POST   /event                       추가 (admin)
PATCH  /event/:slug                 수정 (admin)
DELETE /event/:slug                 삭제 (admin)
GET    /event/:slug/registration    회차 신청자 목록 (admin)
```

### Registration (`/registration`)
```
POST   /event/:slug/registration    신청 (JWT, 정원 트랜잭션)
GET    /me/registration             내 신청 이력 (JWT)
GET    /registration/:token         단일 (본인 또는 admin)
GET    /registration                전체 (admin)
PATCH  /registration/:token         상태 변경 (admin 또는 본인 cancel)
```

### Testimonial (`/testimonial`)
```
GET    /testimonial                 공개 후기 (?q=&page=)
POST   /testimonial                 작성 (회원 또는 anon, anon 시 author_name 직접)
PATCH  /testimonial/:id             수정 (작성자 또는 admin)
DELETE /testimonial/:id             삭제 (admin)
```

### Post / Inquiry / Site / Admin — 3차와 동일 (페이지네이션 + ?q= 검색만 추가)
