## 1. 이름 규칙 — Domain Prefix Tree

### 1-1. 룰

- 단일 도메인 = 짧게: `event`, `member`, `post`, `inquiry`, `site_config`
- 하위 도메인 = `parent_subdomain`: `event_application`, `event_registration`
- 더 깊으면 = `parent_sub_subsub` (필요할 때만): `event_attendance_log`
- DB 테이블 = snake_case / Django 클래스 = PascalCase (`EventApplication`)
- 일관성 위해 명사 유지 (`apply` X, `application` ✅)

### 1-2. 우리 도메인 트리

```
event/
├── event                    회차 (Class instance)
├── event_application        비회원 신청 (1차 진입)
├── event_registration       회원 확정 신청 (정원 트랜잭션)
└── event_testimonial        후기

member/
└── member                   회원 (auth.User 확장)

post/
└── post                     사이트 글 (블로그 + Threads 자동 발행 소스)

inquiry/
└── inquiry                  Q&A 문의

site_config/
└── site_config              운영자 메타 (singleton, id='main')
```

### 1-3. Django 매핑

```python
# apps/events/models.py
class Event(models.Model):
    class Meta: db_table = 'event'

class EventApplication(models.Model):
    class Meta: db_table = 'event_application'

class EventRegistration(models.Model):
    class Meta: db_table = 'event_registration'

class EventTestimonial(models.Model):
    class Meta: db_table = 'event_testimonial'
```

## 2. PK / FK 전략

### 2-1. 분류

| 종류 | 예시 | PK |
|------|------|-----|
| **Domain entity** (의미 있는 ID, URL 노출) | event, member, post, inquiry, site_config | **string slug** |
| **Transaction entity** (누적 row, 의미 없음) | event_application, event_registration, event_testimonial | integer auto-increment + (선택) `token` 컬럼 |

### 2-2. Domain slug 명명 규칙

| 도메인 | prefix | 예시 | 생성 방법 |
|--------|--------|------|---------|
| event | `evt_` | `evt_w1_20260510_1400` | 수동 (admin 입력, ClassEditor) |
| member | `mem_` | `mem_aB3xK9pQ` | **자동 (nanoid 8자)** |
| post | `pst_` | `pst_2026_05_10_supabase_django` | 자동 (제목 → slug, admin 수정 가능) |
| inquiry | `inq_` | `inq_20260510_001` | 자동 (날짜 + 일련) |
| site_config | (singleton) | `main` | 고정 |

**룰**:
- snake_case
- ASCII only (URL 친화)
- prefix 통일 → DB 에서 grep 가능
- **slug 는 한 번 정하면 안 바꾸는 룰** (변경 시 FK 대량 갱신, 위험)

### 2-3. member 의 nickname 분리 (재고 결과)

```python
class Member(...):
    id = models.SlugField(primary_key=True, max_length=20, default=generate_member_id)  # 'mem_aB3xK9pQ'
    nickname = models.CharField(max_length=50, unique=True)  # 'smk6931', 변경 가능
    email = models.EmailField(unique=True, null=True, blank=True)
```

- `id` = 영구 불변 (FK 안전, URL slug 로 직접 노출 X)
- `nickname` = 변경 가능, unique, URL 노출용 (`/u/smk6931`)
- nickname 변경해도 FK (`event_application.converted_member_id`) 갱신 X

### 2-4. event slug 시간 포함 (충돌 방지)

같은 날 두 회차 가능:
- `evt_w1_20260510_1400` (5/10 14시 1주차)
- `evt_w1_20260510_1800` (5/10 18시 1주차)

`week1` → `w1` 짧게. 사람이 슬러그 봐도 의미 즉시 파악.

### 3-1. 관계 요약

- **Event 1 : N EventApplication** (회차 비회원 신청)
- **Event 1 : N EventRegistration** (회차 회원 확정, 정원 제한)
- **Event 1 : N EventTestimonial** (회차 후기)
- **Event N : 1 Curriculum (jsx)** — string slug 약한 연결, FK X
- **Member 1 : N EventApplication** (converted_member_id)
- **Member 1 : N EventRegistration**
- **Member 1 : N EventTestimonial**
- **Member 1 : N Post** (author)
- **Member 1 : N Inquiry** (선택)

### 3-2. 삭제 정책

- `member` 삭제 시 → 그의 application/registration/testimonial 다 **SET NULL** (member_id 만 끊고 author_name 보존)
- `event` 삭제 시 → 그의 application/registration/testimonial 다 **CASCADE** (회차 자체 사라지면 의미 없음)
- 회원이 "내 데이터 완전 삭제 (개인정보 파기)" 요청 시 → 별도 hard delete API (Sprint 5+)

```json
  "next": "/api/v1/event?page=3",

```
- 기본 `page_size=20`, 최대 `?page_size=100`

## 5. 인증 / 권한

### 5-1. JWT (DRF SimpleJWT)

| 토큰 | 만료 | 저장 |
|------|------|------|
| Access | 15분 | 메모리 (JS 변수) — XSS 위험 최소 |
| Refresh | 7일 | HttpOnly Secure SameSite=Lax 쿠키 |

`token_blacklist` 활성화 (logout 시 refresh 폐기).

### 5-2. 권한 클래스 (`apps/accounts/permissions.py`)

```python
class IsAdmin(BasePermission):
    def has_permission(self, request, view):
        return request.user.is_authenticated and request.user.role == 'admin'

class IsOwnerOrAdmin(BasePermission):
    def has_object_permission(self, request, view, obj):
        return obj.member_id == request.user.id or request.user.role == 'admin'
```

## 6. Docker 구성

### 6-1. `docker-compose.yml` (개발)

```yaml
services:
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: vibe
      POSTGRES_USER: vibe
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    ports:
      - "5200:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U vibe"]
      interval: 5s

  back:
    build: ./back
    command: python manage.py runserver 0.0.0.0:8000
    ports:
      - "8200:8000"
    environment:
      DATABASE_URL: postgres://vibe:${DB_PASSWORD}@db:5432/vibe
      DJANGO_SETTINGS_MODULE: config.settings.local
      SECRET_KEY: ${SECRET_KEY}
    volumes:
      - ./back:/app
    depends_on:
      db:
        condition: service_healthy

volumes:
  postgres_data:
```

### 6-2. `back/Dockerfile`

```dockerfile
FROM python:3.12-slim
WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends gcc libpq-dev \
    && rm -rf /var/lib/apt/lists/*
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8000", "-w", "4"]
```

### 6-3. 운영

- Oracle VM 안에서 `docker-compose -f docker-compose.prod.yml up -d`
- Nginx upstream: `location /api/ { proxy_pass http://localhost:8200; }` 추가
- 백업: pg_dump 일일 cron → R2 (또는 Oracle Object Storage)

## 9. 마이그레이션 / Seed

### 9-1. 단계별 전환

1. **Phase 1** (Sprint 1): 백엔드 띄우고 seed 까지. **프런트는 JSON 직접 import 그대로** (병렬 운영)
