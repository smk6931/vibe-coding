# 리포트 3 — 미니홈피 커리큘럼 (4주 vs 6주 vs 8주)

> **이 리포트의 질문**: "1~4주차에 담고 싶은 게 너무 많다. 4주에 다 들어가나? 8주는 너무 길다. 정답은?"

작성일: 2026-04-27
선행 문서: 1차 기획안 §4-3 (미니홈피 컨셉), `OnedayClassCurriculum.jsx` (2시간 단발 교안)

---

## 0. 한 줄 결론

> **6주 × 2시간 × ₩30,000/회 = ₩180,000 패키지가 정답.**
>
> - 4주: 환경세팅·프론트·백엔드·DB·배포·꾸미기를 다 못 담는다. 백엔드가 깊이 안 들어감.
> - 8주: 회당 부담은 작아도 누적 ₩240,000 이상 → 결제 저항 ↑ + 8주 출석률 급락 (업계 통설 4주차 70%, 6주차 55%, 8주차 35%).
> - **6주 = 백엔드/DB 까지 충분히 다루면서, 출석률 60%+, 패키지 가격이 시장 평균 ₩200,000 이하** 의 균형점.

## 1. 사용자가 담고 싶은 컨텐츠 — 인벤토리

> 사용자 메모 정리:
> - 1주차: 프론트 이해, GitHub repo / commit / push

> - 4주차: 미니홈피에 음악 넣기, 미니게임, Vercel 배포, 메모장, 풀링 정리, 일기장, 스케줄 관리, 자기소개/포트폴리오/연락처/홈페이지

이걸 분류하면:

### 1-1. 코어(반드시 들어가야)
- HTML/CSS 첫 페이지 (자기소개)
- React 컴포넌트 (프로필, 카드, 방문자카운터, 다이어리)

- DB (PostgreSQL 또는 SQLite)
- 배포 (Vercel/Render/Netlify, 본인 URL)

### 1-2. 코어+ (있으면 좋음, 없어도 OK)
- 음악 재생 위젯 (HTML5 audio 또는 YouTube embed)
- 별 파티클 / Canvas 애니메이션
- 일기장(다이어리)·메모장·스케줄 관리 — 이건 사실상 같은 CRUD의 변형
- 미니게임 (가위바위보 / 단순 클릭 게임)

### 1-3. 옵션 (별도 강의/멤버십으로 분리하는 게 적합)
- 풀링 기반 자동 정리 (스크래핑, 크롤링) — 난이도 높음 + 법적 이슈
- 캘린더 동기화 (구글 캘린더 API)
- 다중 사용자 인증 (로그인) — 4주차에 도입하면 6주가 8주가 됨
- 알림 / 푸시 — 외부 의존성 큼 (FCM 등)

> **판단**: 코어 7개 + 코어+ 3~4개를 6주에 채우고, 옵션은 "심화 강의" / "멤버십 라이브러리" / "1:1 코칭" 으로 분리.

## 2. 주차 수 결정 — 비교표

| 항목 | 4주안 | **6주안 (권장)** ✅ | 8주안 |
| --- | --- | --- | --- |
| 회당 시간 | 2시간 | 2시간 | 2시간 |
| 패키지 가격 | ₩120,000 | **₩180,000** | ₩240,000 |
| 회당 가격 | ₩30,000 | ₩30,000 | ₩30,000 |
| 코어 7개 커버 | 5개 (CRUD/DB 깊이 부족) | **7개 전부** | 7개 + 옵션 2개 |
| 4주차 결제 저항 | 낮음 (즉시 결제 OK) | 중간 (분할결제 가능) | 높음 (부담) |
| 출석률 (업계 통설) | 70% | 55% | 35% |
| 1주 뒤 retention | 65% | 60% | 25% |
| 운영자 콘텐츠 부담 | 4편 | 6편 | 8편 (한계) |
| 마케팅 슬로건 강도 | "4주 만에" 강력함 | **"6주 안에 미니홈피 + DB + 배포까지"** | "8주 단축 부트캠프" 무거움 |
| 강사 1인 동시 운영 | 월 3기수까지 | **월 2기수가 베스트** | 월 1기수 한계 |
| LTV (멤버십 전환) | 30% | **45%** | 50% (단, 절대 모객수 적음) |

### 2-1. 4주안의 함정
- 코어 7개를 4주에 다 우겨넣으면 회당 부담 ↑ → 첫 회차 빼고 출석률 급락.
- "6주 만에 백엔드까지" 라고 광고하면 신뢰감 ↑ 인데 "4주 만에" 는 "어떻게? 너무 빠르다" 의 의심.
- DB·백엔드를 20분씩만 다루면 결국 "보고만 끝나는" 강의 = 결과물 빈약.

### 2-2. 8주안의 함정
- 8주 결제 부담 ₩24만 = 직장인의 "한 번 결심" 문턱이 분명히 있음 (인지된 부담의 임계).
- 8주차쯤 가면 운영자도 같은 콘텐츠 8회 반복으로 dispatch 무너짐.
- 출석률 통계가 가장 큰 문제. 8주차에 절반 이하 남으면 후기·졸업 발표 섹션이 무력화됨.

### 2-3. 6주안이 정답인 이유
1. **소비자 인지의 "한 분기"** = 6주는 "한 시즌" 으로 자연스러운 감정. (4주는 "한 달", 8주는 "두 달"의 모호한 사이.)
2. **회당 가격 ₩30,000 × 6 = ₩180,000** = 인스타·당근·소모임 시장의 "AI 자기계발 한 분기" 가격에 정확히 맞음. ₩200,000 이하는 결제 저항 적음.
3. **시즌제 운영과 자연스럽게 결합**. S1 (4~5월), S2 (6~7월), S3 (9~10월), S4 (11~12월) 의 4시즌. 시즌 사이 휴식 2~3주.
4. **백엔드/DB까지 충분히 다룰 수 있음**. 4주에 백엔드까지는 "찍먹"이지만 6주는 "이거 진짜 만들었다"의 깊이.

### 2-4. 변형 옵션 (사용자 결정 사항)
- **6+1 (선택형 7주차)**: 6주차 끝나고 "데모데이 발표" 1회를 + ₩0 또는 + ₩20,000 옵션으로. 출석률 낮은 8주안의 약점은 피하면서 "졸업식" 효과를 얻음.
- **3주 × 2 시리즈**: "기초 3주" + "심화 3주" 분할. 첫 3주만 듣고 빠지는 것도 가능. 단 운영 복잡도 ↑.

### 🎓 4주차 — 백엔드 입문 + Express + REST API (router/service)

**한 줄 목표**: 다이어리 데이터를 localStorage가 아니라 "내 컴퓨터의 작은 서버" 에 저장한다. router/service/model/repository 구조를 처음 본다.

#### 학습 목표
- "왜 백엔드인가" 의 직관 (브라우저는 잊어버린다, 서버는 기억한다)
- Express + Node로 GET/POST/DELETE 만들기
- router (URL 분배기) / service (비즈니스 로직) / repository (데이터 접근) 의 분리
- JSON 파일을 임시 DB로 쓰기 (PostgreSQL은 다음 주)

#### 타임라인
| 시간 | 블록 | 내용 |
| --- | --- | --- |
| 0:00–0:10 | 복습 | 3주차 다이어리 — "근데 이거 다른 컴퓨터에선 안 보여요" 의 자연 도입 |
| 0:10–0:30 | 백엔드 개념 | "프론트는 그림, 백엔드는 창고" 비유. router/service/repository 다이어그램 |
| 0:30–0:50 | Express 세팅 | `mkdir backend && cd backend && npm init -y && npm i express`. Claude에 "Express로 GET /api/diary, POST /api/diary 만드는 router/service/repository 구조 만들어줘" |
| 0:50–1:20 | API 만들기 | `routes/diary.js` (URL 분배), `services/diary.js` (로직), `repositories/diary.js` (data/diary.json 읽기/쓰기) |
| 1:20–1:50 | 프론트와 연결 | React에서 `fetch('http://localhost:4000/api/diary')` 로 GET. 다이어리가 서버에 저장되는 것 확인 |
| 1:50–2:00 | 마무리 | 다른 브라우저로 열어서 같은 다이어리가 보이는 것 확인 (= 서버의 의미) |

#### 학생 산출물
- `~/projects/my-hompy-backend/` 별도 repo (또는 monorepo 구조)
- localhost:4000 서버, localhost:5173 프론트 동시 실행
- 다이어리가 `data/diary.json` 에 저장됨

#### 운영 노트
- CORS 첫 만남 — `cors` 미들웨어 빠뜨리면 fetch 안 됨. Claude에 "CORS 에러 났는데 뭐야" 물어보면 바로 답.
- router/service/repository 의 "왜 분리해" 질문이 나옴. **답**: "지금은 작아서 의미 없어 보이지만, 5주차에 PostgreSQL 붙일 때 repository만 바꾸면 service/router는 그대로 — 그게 분리의 진짜 의미." (5주차 미리 예고)
- 빠른 사람: "PUT /api/diary/:id 로 수정도 만들어줘" 추가. 늦은 사람: GET/POST만 동작하면 OK.

#### 과제
- "내 백엔드는 이런 구조다" 다이어그램 손으로 그려서 인스타/카톡방 공유 (선택)

### 🎓 5주차 — PostgreSQL + Claude로 DB 설치부터 마이그레이션까지

**한 줄 목표**: JSON 파일 DB를 진짜 PostgreSQL로 옮긴다. Claude가 처음부터 끝까지 같이 해준다. router/service는 그대로, repository만 바뀐다.

#### 학습 목표
- DB가 뭐고 왜 쓰는지 (영속성 + 검색 + 동시 접근)
- PostgreSQL 로컬 설치 (Mac: brew, Win: 인스톨러)
- Claude로 SQL 첫 작성 (CREATE TABLE, INSERT, SELECT)
- repository 패턴의 진짜 의미 (구현체 교체)

#### 타임라인
| 시간 | 블록 | 내용 |
| --- | --- | --- |
| 0:00–0:10 | 복습 | 4주차 JSON DB의 한계 — "동시에 여러 명이 쓰면 깨진다", "검색이 느리다" |
| 0:10–0:30 | DB 개념 | 표 그림으로 설명. SELECT/INSERT/DELETE의 영어 직관. PostgreSQL이 "그 표를 안전하게 보관하고 빠르게 찾아주는 프로그램". |
| 0:30–1:00 | 설치 | OS별 설치 가이드. Claude에 "Windows에 PostgreSQL 17 설치하는 법 단계별로" / "Mac에 brew로 설치하는 법" |
| 1:00–1:30 | DB 연결 | `npm i pg`. `repositories/diary.js` 를 JSON 파일에서 PG로 교체. Claude에 "위 JSON 코드를 PostgreSQL로 바꿔줘. CREATE TABLE 문도 같이" |
| 1:30–1:50 | 마이그레이션 | `data/diary.json` 의 기존 메모를 INSERT로 옮기기 (Claude에 "이 JSON 데이터를 INSERT 문으로 바꿔줘") |
| 1:50–2:00 | 마무리 | 다이어리 동작 확인. **router/service는 한 줄도 안 바뀜 = 분리의 의미.** |

#### 학생 산출물
- 로컬 PostgreSQL `mh` DB + `diary` 테이블
- 다이어리 데이터가 PG에 저장됨
- repository만 교체되고 router/service는 그대로 (= 학습 목표 체감)

#### 운영 노트
- **이 회차가 가장 어렵다**. 절반 이상이 "PostgreSQL 비밀번호 까먹음" / "포트 5432 점유 충돌" 등으로 막힘. 1:1 시간 가장 많이 듦.
- **사전 준비물 강조**: 사전 카톡으로 "PostgreSQL 설치는 가능하면 강의 전 자기 컴에서 한 번 시도" 권유. 강의 시간엔 디버깅에 집중.
- Claude가 SQL 잘 쓰지만 한국어 컬럼명/한글 인코딩(UTF-8) 이슈 있음. 사전 안내 필요.
- 빠른 사람: "favorites 테이블도 만들어서 카드 데이터도 DB에 옮겨" 보너스. 늦은 사람: SELECT만 동작하면 OK.

#### 과제
- "내 미니홈피 DB 스키마 다이어그램" 손그림 (선택)

### 🎓 6주차 — 꾸미기(음악/파티클) + 미니게임 + Vercel/Render 배포 + 자기 URL

**한 줄 목표**: 미니홈피에 음악 위젯·별 파티클·간단한 미니게임을 붙이고, Vercel(프론트)·Render(백엔드)에 배포한다. **내 URL이 생긴다.**

#### 학습 목표
- HTML5 audio 또는 YouTube embed로 음악 재생
- Canvas 별 파티클 (운영자 미니홈피의 그 효과)
- 가위바위보 같은 단순 미니게임 (state 응용)
- 무료 배포 — Vercel + Render (또는 Railway, Fly.io)

#### 타임라인
| 시간 | 블록 | 내용 |
| --- | --- | --- |
| 0:00–0:10 | 복습 | 5주차 PG 까지 한 명씩 자랑 (이쯤 되면 자존감 ↑) |
| 0:10–0:30 | 꾸미기 | Canvas 파티클 컴포넌트 추가 (운영자 코드 그대로 보고 따라하기) + HTML5 audio로 BGM 추가 |
| 0:30–0:50 | 미니게임 | 가위바위보 컴포넌트 — `useState`, `Math.random`, 결과 메시지. 1주차 자기소개 페이지에 미니홈피의 사이드바로 추가 |
| 0:50–1:20 | Vercel 배포 (프론트) | GitHub 연동 → Vercel import → Deploy. `https://mh-도리.vercel.app` 같은 URL 생성 |
| 1:20–1:50 | Render 배포 (백엔드) | GitHub 연동 → Web Service → 환경변수(DB URL) → Deploy. Render 무료 PG 추가 → 5주차 데이터 마이그레이션 |
| 1:50–2:00 | 졸업 | 한 명씩 1분 자기 URL 자랑. **6주 졸업 인증 카톡방 메시지 + 인스타 태그.** 다음 시즌(스터디/모각작) 안내. |

#### 학생 산출물
- `https://본인이름.vercel.app` (프론트) — **인스타에 자랑 가능**
- `https://본인이름-api.onrender.com` (백엔드)
- 가위바위보 / BGM / 별 파티클 추가된 미니홈피
- 6주 수료 뱃지 (사이트 마이페이지에 자동 발급, 더미 데이터로 시뮬레이트 가능)

#### 운영 노트
- **클라이맥스 회차**. 분위기가 가장 좋아짐. 발표 자원자가 늘어남.
- Render 무료 티어는 15분 idle 시 sleep — 학생들이 "왜 처음엔 느려요" 물음. 미리 안내.
- Vercel + Render 의 이중 배포가 어려운 학생에겐 "Vercel만 + Vercel Functions로 백엔드도" 옵션 제시 (단, 5주차 PG 결합과 약간 안 맞음).
- 졸업 사진 / 인증샷 — 이걸 다음 기수 모집의 가장 큰 자산으로 사용. 동의 받고 인스타·랜딩에 노출.

#### 과제 (졸업 후)
- 자기 URL 카톡방 공유. 다음 시즌 스터디/모각작 추천.

## 8. 첫 시즌(S1) 운영 시뮬레이션

### 8-1. 일정 (예시)
- S1 모집: 5월 1일~14일 (인스타·당근·소모임)
- S1 1기수: 5월 17일~6월 21일 (매주 토요일 15:00~17:00, 6회)
- S1 2기수: 5월 18일~6월 22일 (매주 일요일 15:00~17:00, 6회) — 같은 콘텐츠 토/일 분할
- S1 졸업/데모데이: 6월 28일 (토) 1회 통합

### 8-2. 매출 시뮬레이션
- 1기수 정원 6명 × ₩180,000 = ₩1,080,000
- 2기수 정원 6명 × ₩180,000 = ₩1,080,000
- 합계 S1 매출 = **₩2,160,000** (1.5개월)
- 비용: 강의실 대관 (회당 ₩30,000 × 12회 = ₩360,000) + 마케팅 ₩100,000 = 약 ₩460,000
- 순수익 약 ₩1,700,000

### 8-3. S1 KPI
- 정원 충족률 ≥ 80% (6명 중 5명+)
- 6주 출석률 ≥ 60%
- 졸업 시 "다음 시즌 신청 의향" ≥ 50%
- 후기 캡쳐 ≥ 8건
- 인스타 신규 팔로워 ≥ 200

이 KPI 4개 이상 만족하면 S2 (6~7월) 정상 진행. 미만이면 가격·페르소나·시간대·장소 재검토.
