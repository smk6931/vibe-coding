# 바이브코딩 소모임 + 오프라인 라이트클래스 플랫폼 1차 기획안

작성일: 2026-04-25
작성자: 운영자 본인 + Claude
문서 성격: 사업 + 제품 1차 기획안 (의사결정용 초안)

---

## 0. 한 줄 요약 / TL;DR

- **컨셉**: 수원-서울 중간축에서 운영하는 "2~6인 소수정예 2시간 라이트 클래스 + 지속 운영형 바이브코딩 커뮤니티"
- **이 플랫폼의 가장 강력한 차별점 = "지도+캘린더 기반 전국 바이브코딩 모임 어그리게이션 대시보드"**. 자체 강의만 보여주는 게 아니라 **전국 스터디/모각작/해커톤/사이드프로젝트 모임을 한 화면에 모아주는 "한국 바이브코딩 모임의 메타 검색 엔진"** 역할로 포지셔닝. 이게 SEO·유입·재방문의 핵심.
- **단발 강의(현금흐름) + 어그리게이션 대시보드(트래픽 자석) + 커뮤니티(LTV)** 의 세 축 모델.
- **시장성**: 2026~2028년 약 2~3년간 매우 유효. 단, "바이브코딩 입문"에서 "AI로 사이드프로젝트/1인 SaaS 만들기"로 진화시켜야 길게 간다.
- **배포(사용자 결정)**: 오라클 클라우드 VM(보유 SSH 키, server.ps1 배포) + PostgreSQL 자체 운영. **Phase 1은 백엔드 미연결 — 프론트 + 더미 JSON만으로 프로토타입 검증** 후 OK이면 백엔드 연결. 1차 URL은 IP 직주소(168.x.x.x) 그대로.
- **페이지 구조**: 사용자가 제안한 1~4를 "어그리게이션 대시보드 / 학습 라이브러리 / 커뮤니티 / 마이페이지 / 운영자 어드민"의 5개 허브로 재편.
- **놓친 핵심 기능**: 결제·환불 흐름, 출석/QR 체크인, 알림톡 리마인더, 후기/리뷰, 추천코드(레퍼럴), 어드민 CMS — 이 6개가 비어 있으면 "운영"이 안 됨. 단, 결제는 **단계 진화(Phase 0 계좌송금 → Phase 2 PG)** 가 정답.
- **멤버십은 일단 보류 (사용자 결정)**. 1회 강의 가격 ₩20,000~25,000 (소모임 시장 표준에 맞춤).

## 1. 명칭 정리 ("2시간짜리 짧은 오프라인 강의")

> 사용자 질문: "그 2시간짜리 짧은 오프라인 강의하는걸 머라하냐"

업계에서 가장 통용되는 명칭은 다음과 같다.

| 명칭 | 뉘앙스 | 추천도 |
| --- | --- | --- |
| **원데이 클래스 (One-day Class)** | 클래스101/탈잉식, "단발성 짧은 수업"의 가장 표준 | ★★★★★ |
| 라이트닝 워크숍 / 라이트 클래스 | 짧고 가볍다는 어감 | ★★★★ |
| 쇼트 트랙 / 쇼트 클래스 | IT 부트캠프식 | ★★★ |
| 살롱(Salon) | 소수정예 + 네트워킹 어감 | ★★★ |
| 미니 부트 / 라이트 부트 | 부트캠프 축약형 | ★★★ |

**브랜딩 추천**: 외부 노출(당근/소모임/인스타)에는 **"원데이 클래스"** 또는 **"2시간 라이트 클래스"** 라는 통상 용어를 쓰고, 내부 브랜드명으로는 **"바이브 세션(Vibe Session)"** 같은 자체 네이밍을 따로 부여 → 모임 정체성 강화.

> 예: "이번 주 토요일 — 수원역, 바이브 세션 #12: Cursor로 2시간 만에 내 사이드프로젝트 시드 만들기" (포스터 카피로 바로 사용 가능)

## 2. 시장성 분석 (지금 먹힐까, 얼마나 유효할까)

### 2-1. 거시 흐름
- 2024~2026년: GPT-4/Claude/Cursor/Replit 등으로 **비전공자가 직접 코드를 만지는 임계점** 을 넘은 시기. 국내 검색량/수강자 폭증.
- 클래스101·인프런·패스트캠퍼스의 온라인 강의는 **포화**. 가격 경쟁/할인 경쟁으로 강사 단가 하락.
- 반대로 **오프라인 소수정예 모임**은 공급이 매우 적음. (위코드/패스트캠퍼스는 대형·고가·장기, 살롱드코딩/네이버카페는 비정기·무료·운영자 번아웃 잦음)
- **빈 시장**: "비전공자/주니어가 부담 없이 가는 2시간짜리 + 같은 동네 사람들 + 진짜 손에 잡히는 결과물(=내 프로젝트)" 의 결합.

### 2-2. 사용자 제안의 강점
1. **2인 정원**의 심리적 진입장벽 ↓ (1대다 강의의 부담감 회피)
2. **수원-서울 사이 지역 한정** → 타겟이 분명해 마케팅 효율 ↑
3. **단발 결제 → 커뮤니티/스터디로 이어지는 funnel** → LTV 확보 가능
4. **운영자가 코더 본인** → 진정성/콘텐츠 발화량 모두 유리
5. **(가장 중요) 지도+캘린더 어그리게이션 대시보드** — 자체 강의만 띄우는 게 아니라 전국 바이브코딩 스터디/모각작/사이드프로젝트 모임을 한 화면에 모아주면, 강의 신청자 외 **"이번 주에 어디서 뭐 하는지 보러 오는" 비강의 트래픽** 을 빨아들임. 한국에 이걸 깔끔하게 모아주는 사이트가 사실상 없다 (소모임/당근/디스콰이엇은 분산됨). 검색 키워드 "바이브코딩 스터디", "수원 코딩 모임", "AI 코딩 스터디 서울"의 자연 유입을 장악할 수 있는 SEO 자산이 된다.

### 2-3. 리스크 / 약점
| 리스크 | 설명 | 완화책 |
| --- | --- | --- |
| 강사 1인 의존 | 강사가 아프거나 바쁘면 매출 0 | 보조강사/조교 트랙, 영상강의/전자책으로 비강사 매출 채널 확보 |
| 도구 변화 빠름 | Cursor, Claude, GPT의 UI/기능이 6개월마다 변함 → 콘텐츠 갱신 부담 | 콘텐츠를 "도구 사용법"보다 "사이드프로젝트 만드는 사고법"으로 잡으면 수명이 길어짐 |
| 노쇼/환불 | 오프라인 단발 클래스의 고질병 | 부분 선결제 + 출석 시 일부 환급(=리워드 구조), QR 체크인, 알림톡 리마인더 |
| 단발 강의 LTV 낮음 | 한 번 듣고 끝남 | 멤버십/스터디로 묶기, 후속 강의 트랙(입문→실전→배포) 설계 |
| 지역 수요 한계 | 수원-서울 중간 = 인구 풀 자체는 큼 (광교/수원역/사당/강남 모두 접근 가능)이지만 정기 회차를 채우려면 매주 신규 유입 필요 | 인스타/당근 콘텐츠 마케팅 + SEO 블로그 (장기 자산) |

### 2-4. 유효 기간 전망
- **2026~2028년 (강력)**: "바이브코딩 입문" 키워드 자체로도 충분히 수요 흡수 가능.
- **2028년 이후 (전환 필요)**: AI 코딩이 "당연한 도구"가 되면 입문 수업의 매력 하락. 이때는
  - "AI로 1인 SaaS 만들기"
  - "AI 자동화로 본업 부수입 만들기"
  - "비개발자 사이드프로젝트 → 실제 매출 만들기"
  로 콘텐츠 축을 옮겨야 함. 즉 **"바이브코딩"이라는 단어보다 "AI로 작은 결과물 만드는 능력"이 본질** 이라는 방향성 유지가 핵심.

> **결론**: 지금 시작하기에 타이밍 매우 좋음. 단, 처음부터 "강의 하나 팔기"가 아니라 **"커뮤니티+학습라이브러리+멤버십"의 토대를 동시에 깔아야** 1년 뒤에도 살아 있다.

## 3. 배포 전략 — 사용자 결정 반영

### 3-0. 사용자 결정 사항 (확정)
- **단일 인프라: 오라클 클라우드 VM** (보유 SSH 키 + 자체 `server.ps1` 배포 스크립트 사용).
- **PostgreSQL 자체 운영** (Supabase 미사용).
- **URL은 1차에 IP 직주소(예: `http://168.x.x.x/`) 그대로 사용**, 도메인은 추후.
- **Phase 1은 백엔드 미연결**. 프론트엔드 + 더미 JSON만으로 프로토타입을 띄워 화면·UX 검증을 먼저 한 뒤 OK면 백엔드 연결.

### 3-1. 합의된 아키텍처 (단순 단계 진화형)

| 단계 | 프론트 | 백엔드 | DB | 비고 |
| --- | --- | --- | --- | --- |
| **Phase 1 (프로토타입)** | Next.js 정적 빌드(또는 SSG) | **없음** | 없음 (`/public/data/*.json` 더미) | 오라클 VM에 정적 호스팅. IP URL 그대로. server.ps1로 배포 |
| **Phase 2 (백엔드 연결)** | Next.js | Node/Express 또는 Next API | PostgreSQL (오라클 VM 동일 박스) | 회원가입·이벤트 CRUD·관리자 화면 연결 |
| **Phase 3 (스케일/도메인)** | Next.js | 동일 | 동일 + Redis 캐시 | 도메인·SSL(Let's Encrypt 또는 Cloudflare 프록시) |
| **Phase 4 (확장)** | 동일 | + WebSocket 채팅(Socket.io) | + 파일 스토리지(R2 또는 VM 디스크) | 채팅·영상 |

### 3-2. 단일 박스 운영 시 주의점 (1인 운영자 체크리스트)
- **백업**: PostgreSQL `pg_dump` 일일 자동 → S3/R2 또는 별도 서버. **이걸 안 하면 1년 내 한 번은 운다.**
- **방화벽**: Oracle Cloud Security List + ufw 둘 다 열어줘야 함 (오라클 자주 빠뜨리는 함정).
- **SSL**: 도메인 붙는 시점에 Caddy 또는 Nginx + Let's Encrypt 자동갱신. 무료.
- **모니터링**: 최소한 `uptime kuma` 같은 셀프호스팅 모니터를 같은 박스 또는 별도 무료티어에 띄워 다운 알림.
- **배포 스크립트**: 본인이 가진 `server.ps1`을 표준화 — `git pull → build → pm2 reload` 정도의 멱등 스크립트로 만들어 두면 새벽 배포도 안전.

### 3-3. Vercel을 안 쓰는 trade-off (인지하고 가자)
- **잃는 것**: 무료 글로벌 CDN, PR 미리보기 자동 배포, 자동 SSL, 함수 자동 스케일.
- **얻는 것**: 비용 0(오라클 Always Free), DB/백엔드/파일/채팅을 한 박스에서 자유 운영, 외부 서비스 종속성 없음, 본인의 인프라 지식 자산화.
- **결론**: 운영자가 SSH/리눅스 운영에 익숙하다는 전제 하에 합리적 선택. 단, 위 3-2의 백업·방화벽·SSL 자동화는 사이트 출시 직후 1~2주 내에 반드시 세팅해야 한다.

## 4. 페이지 / 정보구조(IA) 재설계

### 4-1. 사용자 제안의 문제 진단
사용자가 제안한 4페이지(강의일정 / 학습자료 / 포트폴리오 / 스터디일정)는
- "강의일정"과 "스터디일정"이 분리되어 있어 회원이 두 곳을 봐야 함 (한 화면에 모이는 게 자연스러움)
- "전자책 / 영상강의" 분리 고민은 **분리하지 말고 "주제 기반 라이브러리"로 통합** 하는 게 사용자 행동에 맞음 (사용자는 "Cursor 입문"을 찾지, "영상이냐 글이냐"로 찾지 않음)
- 운영자 화면(어드민), 결제, 마이페이지, 후기 등 **운영에 필수적인 페이지가 빠짐**

### 4-2. 권장 IA — 5개 허브 + 3개 보조

```
[공개]
  ① 홈/랜딩                   → 비회원 전환 (다음 클래스 CTA, 후기, 모임 소개)
  ② 어그리게이션 대시보드     → ★핵심 차별점★
                                 자체 강의(수원-서울) + 전국 스터디/모각작/사이드프로젝트
                                 = 지도 + 캘린더 + 리스트 3-뷰 토글
  ③ 학습 라이브러리           → 전자책 + 영상강의 + 블로그 글 통합 (주제/난이도 필터)
  ④ 커뮤니티 허브             → 회원 포트폴리오 쇼케이스 + 자유 게시판 + 회원 찾기

[로그인 후]
  ⑤ 마이페이지                → 내 수강내역 / 내 포트폴리오 / 결제내역 / 출석률·뱃지

[운영자 전용]
  ⑥ 어드민 대시보드           → 회원/매출/노쇼/콘텐츠 CMS/이벤트 승인(어그리게이션 등록 검수)

[보조]
  ⑦ FAQ / 환불정책 / 약관
  ⑧ 블로그 (SEO 유입용 — /blog/...)
  ⑨ 결제·신청 흐름 (이벤트 카드에서 진입하는 모달/스텝)
```

### 4-3. 각 허브 상세

#### ① 홈/랜딩 (비회원 전환 페이지)
- **목적**: 당근/인스타에서 들어온 비회원을 30초 안에 "다음 클래스 신청" 또는 "회원가입"으로 전환.
- 구성: 히어로(다음 클래스 D-day 카운트다운) → 모임 소개 1문단 → 후기 캐러셀 → "이 모임으로 만든 작품들" 3~6개 → 가격/일정 CTA.
- 효과: 첫 방문 전환율을 좌우. 채팅·게시판 같은 무거운 요소 절대 넣지 않음.

#### ② 어그리게이션 대시보드 ★핵심 차별점★ (사용자 제안 1번 + 4번 + 외부 모임 통합)

> 이 페이지는 단순 "이벤트 캘린더"가 아니라 **"한국 바이브코딩/AI코딩 모임의 메타 검색 엔진"** 으로 포지셔닝한다. 자체 강의는 그 위에 얹혀 있는 한 종류일 뿐.

- **표시 대상 (이벤트 종류)**:
  - 자체 원데이 클래스 (수원-서울 라인) — `internal`
  - 자체 스터디/모각작/데모데이 — `internal`
  - **외부 모임 (전국)** — 소모임/당근/디스콰이엇/페스타/이벤터스/오프 등에서 운영자·회원이 등록(또는 봇 크롤링) — `external`
- **3-뷰 토글**:
  1. **지도 뷰**: 카카오맵/네이버맵에 핀, 수원-서울 라인은 자체 마커 색상으로 강조, 전국 외부 모임은 다른 색.
  2. **캘린더 뷰**: 월/주 단위, 색상으로 유형 구분(자체 강의/스터디/외부).
  3. **리스트 뷰**: D-day 정렬, 카드형 (제목·일시·장소·주최·잔여석·"신청하러 가기" CTA).
- **필터**: 지역(수도권/충청/영남/호남/제주), 유형(원데이 클래스/스터디/모각작/해커톤/세미나), 난이도, 가격대, 출처(`internal`/`external`).
- **외부 이벤트 등록 흐름**:
  - **운영자 수동 등록(Phase 1~2)**: 어드민에서 "외부 모임" 추가 → 검수 후 노출. 처음 100건은 손으로 채워 데이터셋부터 만든다(SEO 자산이 됨).
  - **회원 제보(Phase 2)**: 로그인 회원이 "이런 모임도 있어요" 폼으로 제출 → 운영자 승인 → 등록한 회원에게 포인트/뱃지.
  - **크롤러(Phase 3+, 선택)**: 소모임/당근 공개 페이지의 약관을 검토해서 가능 범위에서. **저작권·약관 위반 위험 있어 신중**, Phase 3에서 다시 판단.
- **각 이벤트 카드의 행동 분기**:
  - `internal` → 사이트 내 신청·결제로.
  - `external` → 원본 모임 페이지로 outbound (소모임/당근 링크). UTM 태그 붙여서 어떤 모임이 우리 사이트에서 트래픽을 받는지 측정.
- **왜 이게 강력한가 (효과)**:
  1. **SEO 점령**: "수원 코딩 모임", "서울 바이브코딩 스터디", "AI 코딩 모각작" 같은 롱테일 키워드가 자체 강의보다 훨씬 검색량이 큼. 이 페이지가 그걸 다 흡수.
  2. **트래픽 자석**: 강의 결제 의향이 없어도 "이번 주 뭐 있지?" 보러 오는 비강의 트래픽이 들어옴 → 그중 N%가 자체 강의 카드 노출 → 전환.
  3. **재방문 트리거**: 캘린더는 매주 갱신되는 콘텐츠라 자연스러운 재방문 동기를 만든다.
  4. **시장 권위**: "한국 바이브코딩 모임 정보는 여기에 다 있다"는 포지션은 한 번 잡으면 후발주자가 뒤집기 어려운 자산이 된다.
  5. **광고/제휴 자산**: 외부 모임 운영자가 자기 모임 노출을 원하게 되면 **유료 프로모션(Sponsored)** 슬롯도 가능 (Phase 4+).

#### ③ 학습 라이브러리 (사용자 제안 2번 통합·확장)
- **권장**: 전자책 / 영상강의 / 블로그 글을 **하나의 페이지** 에서 카테고리·태그·난이도 필터로 통합. 콘텐츠 카드에 매체 아이콘만 표시.
- **이유**: 사용자는 "Cursor 입문"을 찾지 "영상이냐 글이냐"로 찾지 않는다. 분리하면 회원은 두 페이지를 옮겨다녀야 하고, 운영자는 두 페이지를 채워야 하는 부담이 두 배가 된다.
- 카테고리 예: 입문 / Cursor·Claude 활용 / Next.js 배포 / 사이드프로젝트 사례 / 트러블슈팅.
- 콘텐츠 단위: 무료 / 회원 전용 / 멤버십 전용 / 결제(개별 판매)로 권한 분기.
- **효과**: SEO 유입(블로그 글) → 회원가입(회원전용 글) → 멤버십(라이브러리 무제한) 의 자연스러운 funnel 형성.

#### ④ 커뮤니티 허브 (사용자 제안 3번 + 회원 디렉토리 + 게시판)
- 탭 구조: **쇼케이스(포트폴리오) / 자유게시판 / Q&A / 회원 디렉토리**.
- 쇼케이스: 회원이 만든 사이드프로젝트/배포 사이트/깃허브를 카드형으로 노출 → 좋아요·댓글.
- 회원 디렉토리: 자기소개, 관심 스택, "스터디원 찾는 중" 토글 → 같은 관심사끼리 매칭.
- **효과**: 단순 게시판이 아니라 "내가 한 걸 자랑할 곳" 이 있으면 retention이 폭발한다. 인스타에 자랑하기 + 모임에서 자랑하기의 두 축이 모두 자연스러운 콘텐츠 마케팅이 된다.

#### ⑤ 마이페이지
- 내 수강내역(QR 입장권 포함), 결제·환불, 내 포트폴리오 관리, 내 게시글, 출석률/뱃지, 알림 설정(알림톡/이메일).
- **효과**: 1인 운영자에게 가는 "내 환불 어디까지 됐어요?" 같은 카톡 문의를 셀프서비스로 흡수.

#### ⑥ 어드민 대시보드 (사용자가 빠뜨린 핵심)
- **반드시 필요**. 없으면 회원 등록·매출 정산·노쇼 처리·콘텐츠 발행을 모두 DB 직접 만지거나 엑셀로 해야 함 → 1년 안에 번아웃.
- 핵심 화면: 매출/예약 현황, 회원 검색·정지·환불, 클래스 등록·정원 조정, 콘텐츠 CMS, 문의함, 출석체크 콘솔.

### 4-4. 채팅을 어디에 둘까?
- **추천**: "전체 단톡" 하나가 아니라 **이벤트(클래스·스터디)별 채널** 로 자동 생성. 클래스 신청 → 자동으로 그 클래스 채널 가입.
- 0~3개월은 **카카오 오픈채팅으로 외주화** (개발 0). 회원 매뉴얼에 카톡방 링크 자동 발송.
- 3개월 이후 자체 채팅(Socket.io @ 오라클)으로 내재화. 이렇게 단계적으로 가야 비용·기능·운영부하가 모두 합리적.

## 5. 사용자가 빠뜨린 "운영에 꼭 필요한" 기능 6가지

> 강의·콘텐츠·커뮤니티 페이지만으로는 "모임이 굴러가지" 않는다. 아래는 1차 출시에 반드시 들어가야 운영이 무너지지 않는 핵심 기능들.

### (A) 결제 + 환불 흐름 — **가장 중요**
- 토스페이먼츠 / 포트원(아임포트) 연동.
- 부분 선결제(예약금) + 출석 시 잔액 결제 옵션 → **노쇼 방지의 가장 큰 레버**.
- 자동 영수증/세금계산서 발행, 환불 정책 페이지 + 자동 환불 처리.
- 효과: 운영자가 송금 확인·환불 카톡으로 시간 녹이는 것을 90% 차단.

#### (A-extra) 결제 방식 선택지 — 처음엔 계좌 송금? 처음부터 PG?

> **배경**: 소모임 앱의 기존 "바이브코딩 스터디" 모임들이 회당 22,000~25,000원 / 6~8명 정원 / **계좌 송금** 으로 운영 중. 이 방식을 그대로 따라할지, 처음부터 자동결제(PG)를 박을지의 의사결정.

##### 옵션 1 — 계좌 송금 (오프라인 입금)만 사용
- **장점**:
  - 개발 비용 0, 즉시 시작 가능.
  - PG 수수료 0% (토스페이먼츠는 약 3.0~3.3% + VAT). 회당 25,000원의 3.3%는 825원이지만, 월 50만원 미만 매출에서는 누적 부담이 의외로 큼.
  - 사업자등록·통신판매업 신고·PG 심사(보통 1~2주) 부담 없음 → **첫 강의를 가장 빨리 팔 수 있음**.
  - 소모임 앱 사용자에게 익숙한 방식이라 거부감 0.
- **단점**:
  - 입금 확인이 수동 → 운영자 카톡 시간 녹음.
  - 노쇼 시 환불 협의/송금이 모두 수동.
  - 매출/회원 데이터가 자체 DB에 안 쌓이고 카톡·통장 거래내역에 흩어짐 → 자산화 안 됨.
  - 자동 영수증/세금계산서 X → 사업자등록 후엔 별도 정리 필요.

##### 옵션 2 — 처음부터 토스페이먼츠/포트원 PG 도입
- **장점**: 자동 결제·환불·영수증·매출 통계. 데이터 자산화. 신뢰감.
- **단점**:
  - 사업자등록 + 통신판매업 신고 + PG 심사로 출시 1~2주 지연.
  - 사이트 만들기 전엔 의미 없음 → 이 옵션을 고르면 사이트 출시 전엔 강의를 못 팜.
  - 매출 미미한 초기에 정산/세무 부담만 늘어남.

##### 옵션 3 — **단계 진화 (강력 추천 ✅)**
> 한 마디로: "**처음엔 손으로, 패턴이 보이면 자동화**". 자동화는 운영 데이터가 있어야 의미 있고, 운영 데이터는 손으로 몇 번 돌려본 다음에야 잡힌다.

| 단계 | 기간 | 결제 방식 | 사이트 | 신청 도구 |
| --- | --- | --- | --- | --- |
| **Phase 0** | 0~1개월 (1~3회 강의) | **소모임 앱 + 계좌 송금** | 없음 | 소모임 앱·당근·인스타 DM |
| **Phase 1** | 1~3개월 (자체 사이트 MVP) | **토스 송금 링크 / 토스아이디 + 신청 폼** (PG 미도입, "신청은 사이트, 입금은 계좌"의 반자동) | Vercel + Supabase | 사이트 신청 폼 |
| **Phase 2** | 3~6개월 (PG 전환) | **토스페이먼츠 / 포트원 정식 연동** + 자동 영수증 + 자동 환불 + 부분 선결제(예약금) | 동일 | 동일 |
| **Phase 3** | 6개월~ | + 멤버십 정기결제, 알림톡 자동 리마인더, QR 출석 자동 → 뱃지/멤버십 할인 트리거 | 동일 | 동일 |

**Phase 2(PG 도입) 트리거 기준선** — 이 중 하나라도 만족하면 자동화로 전환:
- 월 매출 **100만원 이상** 안정적
- 월 신청자 **40~50명 이상**
- **노쇼율 15~20%** 넘어서 수동 환불·리마인더 카톡이 운영자 시간을 잠식
- 회원이 "카드결제 안 되냐"고 **반복적으로 묻는** 경우 (전환율에 영향)

**선결제 정책은 옵션 1·3-Phase1 단계에서도 손으로 적용 가능**:
- "신청 시 전액 송금 → 강의 24h 전 취소는 100% 환불, 24h 이내 취소는 50% 환불, 노쇼는 0%" 같은 정책을 폼/안내문에 명시.
- 이 정책 자체가 수동이어도 노쇼율을 30~40% 줄임. PG가 없어도 정책의 효과는 본질적으로 동일.

##### 추천 ✅ — **옵션 3 (단계 진화)**
**이유**:
1. **수요 검증이 우선**. 소모임 캡쳐에 보이듯 이미 회당 6~8명 정원이 다 차는 시장이라 수요는 거의 확실하지만, **본인의 강의·페르소나·가격이 그 수요를 잡는지**는 1~2회 돌려봐야만 안다. 이 단계에서 PG 자동화는 과잉 투자.
2. **사이트 짓는 시간 절약**. Phase 0에서 소모임 앱이 무료로 호스트(참석자 명단·댓글·알림) 역할을 해주므로 사이트 0줄로 강의 1~2회를 굴리며 컨텐츠/노쇼율 데이터를 확보할 수 있다.
3. **Phase 1의 "토스 송금 링크 + 신청 폼" 단계가 가성비의 핵심**. PG 없이도 신청·결제·확인을 사이트로 통합할 수 있고, 통장 거래내역과 신청 폼만 매칭하면 된다. PG 수수료 0%, 사업자/통신판매 부담 0, 그러나 데이터는 본인 DB에 쌓임.
4. **PG는 "수수료를 낼 가치가 있을 때" 도입**. 위 트리거 기준선을 만족하기 전엔 자동화 ROI가 마이너스. 먼저 매출이 자동화 비용을 정당화할 수준에 도달하고 나서 도입한다.

##### 비추천 ❌
- **옵션 2 (처음부터 PG)**: 사업자등록·심사 대기로 첫 강의가 1~2주 늦어지는 게 가장 큰 손해. "결제가 자동이라 세련돼 보인다"는 건 회당 6~8명 모이는 단계에선 의미 없는 가치다.

##### 최종 운영 룰 (Phase 0~1 동안)
- 신청 폼에서 본인 명의 + 입금자명 받기 → 통장 거래내역과 매칭.
- "입금 확인 후 24h 내 카톡으로 입장권 안내" 자동응답 멘트 고정.
- 환불 정책 명시(위 24h룰).
- **신청 → 입금 → 입장권 발송**의 3단계는 노션/스프레드시트로 관리해도 충분. 회당 8명 × 월 3회 = 월 24명까지는 손으로 돈다.


### (B) 출석 / QR 체크인
- 신청 완료 시 QR 입장권 발행 → 강사가 폰 카메라로 스캔 → 출석 자동 기록.
- 출석 데이터는 → 뱃지/명예회원/멤버십 할인 트리거로 재활용.
- 효과: 노쇼율을 데이터로 보고 운영 개선. 회원에게는 "출석률 80% 이상 멤버십 할인" 같은 게이미피케이션.

### (C) 알림톡 / 이메일 리마인더
- 카카오 알림톡(비즈니스 메시지) 또는 SMS — 강의 24시간 전 / 2시간 전 / 당일 아침.
- 노쇼율을 통계상 30~50% 줄임 (업계 통설).

### (D) 후기 / 리뷰
- 강의 종료 후 24시간 내 1줄 + 별점 후기 자동 요청.
- 홈/랜딩에 자동 노출 → 신규 전환율 ↑, 구글 검색 SEO에도 도움.

### (E) 추천 코드 / 레퍼럴
- "친구 데려오면 둘 다 1만원 할인" 식의 추천 코드.
- 1인 운영자가 광고비를 쓰지 않고 유입을 늘릴 수 있는 **가장 큰 단일 레버**. 인스타 광고보다 효율 높음.

### (F) 어드민 CMS + 문의함
- 위 ⑥ 어드민과 동일. 문의함은 "환불 가능한가요?" 같은 반복 질문을 FAQ로 승격해 자동응답화.

## 6. 추가 수익 다각화 / 재미 / 지속 유입 전략

### 6-1. 수익 채널 (현재 1축 → 권장 5~6축)

| 채널 | 우선순위 | 메모 |
| --- | --- | --- |
| 원데이 클래스 (현재) | ★★★★★ | 현금흐름의 시작점 |
| **월 멤버십 구독** | ★★★★★ | LTV의 핵심. 라이브러리 무제한 + 스터디 우선예약 + 멤버 전용 채널 + 데모데이 우선석. **반드시 1차 출시에 포함** |
| 영상강의 / 전자책 판매 | ★★★★ | 오프라인 강의를 영상화 → 비강사 매출 (강사 시간이 비는 시간에도 수익) |
| 1:1 코칭 / 사이드프로젝트 멘토링 | ★★★★ | 단가 높음 (회당 5~10만원). LTV 높은 회원만 진입 |
| 기업 출강 (B2B) | ★★★★ | 단가 가장 큼 (회당 50~200만원). 6개월 이후 후기·포트폴리오 쌓이면 진입 가능 |
| 제휴 수수료 | ★★★ | Cursor / Claude Pro / Vercel Pro / 도서 추천 링크. 비강제, 자동수익 |
| 굿즈 / 스티커 | ★★ | 브랜드 충성도 측면. 직접 수익보다 마케팅 자산 |

> **출시 1차 권장 조합**: 원데이 클래스 + 월 멤버십 + 전자책 판매. 이 3축이면 강사가 못 뛰는 주에도 매출 0이 안 된다.

### 6-2. 재미 / 유입 메커니즘

1. **월 1회 "바이브 데모데이"** — 회원이 만든 사이드프로젝트 5분 발표 × 4명. 인스타/유튜브 라이브로도 송출. 발표자에게는 멤버십 1개월 무료 같은 보상.
   - 효과: 회원에겐 "내 결과물을 자랑할 무대", 외부엔 "여기 모임에선 진짜로 뭔가 만들어지네" 라는 사회적 증명.
2. **30일 챌린지** — "주말마다 사이드프로젝트 한 개씩 4주" 식. 인스타 인증 해시태그 → 자연 유입.
3. **명예 뱃지 / 레벨 시스템** — 출석률, 포트폴리오 공유, 후기 작성에 따라 "메이커 / 시니어 메이커 / 멘토" 호칭 부여. 멘토 등급은 신규 회원에게 답글 달면 활동 점수 ↑.
4. **콘텐츠 마케팅** — 인스타 릴스(강의 하이라이트 30초) + 유튜브 숏츠 + 블로그 SEO("수원 코딩 모임", "비전공자 Cursor 입문" 같은 롱테일 키워드). 블로그는 학습 라이브러리와 같은 DB → 콘텐츠 한 번 만들고 두 곳에서 활용.
5. **추천 코드** — 위 5(E) 그대로.
6. **시즌제 운영** — "S1 (4~6월) 입문 시즌", "S2 (7~9월) 사이드프로젝트 시즌" 식. 시즌 종료 시 데모데이로 마무리. 모임에 리듬감을 줘서 "끝나는 느낌" 없이 다음 시즌 기대를 만든다.

## 7. 1차 출시 우선순위 (MVP 범위)

### Phase 0 — 0~4주 ("사이트 0줄, 수요 검증")
- [ ] 소모임 앱 + 당근 + 인스타로 모임 개설, 첫 1~2회 강의 오픈
- [ ] 신청은 소모임 앱/구글 폼, 입금은 계좌 송금
- [ ] 환불 정책(24h룰) 안내문 고정
- [ ] 신청·입금·입장권 발송을 노션/스프레드시트로 손 운영
- [ ] 후기는 카톡·인스타 DM으로 받아 캡쳐 보관 (랜딩에 쓸 자산)
- 의도: **사이트 짓기 전 페르소나·가격·노쇼율을 손으로 학습**.

### Phase 1 — 4~10주 (자체 사이트 MVP, "PG 없이 사이트로 통합")
- [ ] 홈/랜딩 (다음 클래스 CTA + Phase 0 후기)
- [ ] 이벤트 캘린더 (강의·스터디 통합)
- [ ] 회원가입/로그인 (소셜 로그인 — 카카오·구글)
- [ ] 신청 폼 + 토스 송금 링크/계좌번호 (자동 결제 X)
- [ ] 환불 정책 페이지 + 입금 확인 후 입장권(이미지/QR) 자동 발송
- [ ] 어드민 대시보드 (회원·신청 현황·입금 체크박스)
- [ ] 카카오 오픈채팅 링크로 채팅 외주화
- 배포: Vercel + Supabase
- 의도: **사이트로 신청·관리는 자동화, 결제는 아직 손으로**. 본인 DB에 회원·신청 데이터가 쌓이기 시작.

### Phase 2 — 10~16주 (PG 전환, "결제·알림·출석 자동화")
- [ ] **토스페이먼츠 / 포트원 정식 연동** (사업자등록·통신판매업 신고 선행)
- [ ] 부분 선결제(예약금) + 자동 환불
- [ ] 알림톡/이메일 리마인더 (24h / 2h)
- [ ] **QR 입장권 + 강사 폰 스캔 출석 체크**
- [ ] 후기/별점 시스템
- 의도: **트리거 기준선(월 매출 100만원/월 신청자 40명/노쇼율 15~20%) 넘는 시점에 자동화**. 자동화 ROI가 양수가 된 다음에 박는다.

### Phase 2 — 6~14주
- [ ] 학습 라이브러리 (전자책·영상·블로그 통합, 주제 필터)
- [ ] 커뮤니티 허브 (쇼케이스 + 게시판 + 회원 디렉토리)
- [ ] 마이페이지 강화 (포트폴리오 관리·뱃지)
- [ ] 후기/별점 시스템
- [ ] 추천코드(레퍼럴) — 가장 큰 유입 레버
- [ ] 월 멤버십 구독 결제
- 의도: **단발 매출 → 지속 매출(멤버십)** 전환.

### Phase 3 — 16주~ (커뮤니티·자산화)
- [ ] 학습 라이브러리 (전자책·영상·블로그 통합, 주제 필터)
- [ ] 커뮤니티 허브 (쇼케이스 + 게시판 + 회원 디렉토리)
- [ ] 마이페이지 강화 (포트폴리오 관리·뱃지)
- [ ] 추천코드(레퍼럴) — 가장 큰 유입 레버
- [ ] 월 멤버십 구독 결제

### Phase 4 — 24주~ (수익 다각화)
- [ ] 자체 채팅(Socket.io @ 오라클 클라우드)
- [ ] 영상강의 호스팅(Mux/Bunny.net)
- [ ] 1:1 코칭 예약 시스템
- [ ] 기업 출강 랜딩 (B2B)
- [ ] 데모데이 정기화 + 라이브 송출
- 의도: **자산화 + 수익 다각화**.

## 10. 한 줄 결론

> **사이트를 짓기 전에, 첫 클래스를 한 번 팔아봐라.** 사이트는 "이미 굴러가는 모임"의 운영 부담을 줄이는 도구지, 모임을 만들어 주는 마법이 아니다. 첫 1~2회를 구글 폼+토스 송금으로 돌리면서 페르소나·가격·노쇼율을 손으로 익히고, 그 데이터를 기반으로 본 기획안의 MVP를 6주 안에 배포하는 순서가 이 사업의 가장 빠른 길이다.
