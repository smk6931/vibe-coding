## 0. 한 줄 요약

- **만든다**: Next.js 정적 빌드 + `/public/data/*.json` 더미 데이터 + 화면 4개(랜딩/대시보드/이벤트 상세/About).
- **안 만든다**: 회원가입·로그인·결제·DB·관리자·채팅·후기 — **전부 다음 단계**.
- **목적**: "어그리게이션 대시보드"와 "랜딩 → 신청 CTA" 흐름이 **실물로 봤을 때 매력적인지** 를 본인 + 5~10명에게 보여주고 피드백 받는다.
- **배포**: 오라클 VM에 정적 호스팅 (Nginx 또는 `serve`), URL은 `http://168.x.x.x/` 형태 그대로. `server.ps1`로 SSH·rsync·재시작 자동화.
- **기간**: **5~10일** 안에 첫 버전. 길게 끌면 안 됨 — 어차피 더미라 완성도가 의미 없다.

- 실제 결제 게이트웨이 연동 (CTA는 더미 모달)
- 실제 채팅 (카톡 오픈채팅 링크로)
- 실시간 데이터 (모두 정적 JSON)

### 1-3. 채택한 핵심 검증 가설 (이 데모로 답을 얻고 싶은 것)
1. **어그리게이션 대시보드의 첫인상이 "오 이거 한국에 없던 거다" 인가?** (강함의 정도)
2. **지도 vs 캘린더 vs 리스트** 중 사용자가 자연스럽게 어떤 뷰를 먼저 쓰는가?
3. **외부(`external`) vs 자체(`internal`) 이벤트 색 구분** 이 한눈에 들어오는가? 자체 강의 카드의 클릭률이 외부보다 의미있게 높은가?
4. 랜딩의 "다음 클래스 D-day"가 대시보드보다 매력적인가, 아니면 대시보드를 메인으로 두는 게 나은가?

> 이 4가지에 대한 **5~10명 인터뷰** 가 1차 데모의 진짜 산출물이다. 코드는 부산물.

### 3-4. 데이터 시드 작업
- 자체 이벤트 더미 **3건**, 외부 이벤트 더미 **10~15건** (소모임 캡쳐의 모임들로 시드).

- `lat/lng`는 카카오맵에서 주소 입력 → 좌표 복사. 손으로 30분 내.

### 3-5. 추가 더미 JSON (확장 범위에서 추가)
- `library.json` — 학습 콘텐츠 약 12건. 필드: `id, format(book/video/blog), title, summary, category, level, access(free/member/paid), thumbnail, body`.
- `showcase.json` — 회원 포트폴리오 약 8건. 필드: `id, memberId, title, demoUrl, repoUrl, tags, likes, comments, thumbnail`.
- `board.json` — 자유 게시판 약 10건. 필드: `id, memberId, title, body, createdAt, comments, likes`.
- `qa.json` — Q&A 약 6건. 필드: `id, memberId, title, body, answerCount, resolved, createdAt`.
- `members.json` — 회원 디렉토리 약 12명. 필드: `id, nickname, avatar, intro, stack[], lookingForBuddy(bool), region`.
- `me.json` — 마이페이지용 단일 사용자. 필드: `enrollments[], payments[], portfolios[], attendanceRate, badges[], notifications`.
- `admin.json` — 어드민 통계. 필드: `revenue{daily,weekly,monthly}, reservations{open,filled}, noShowRate, memberCount, inquiriesPending, classCalendar[]`.

## 4. 기술 스택 (1차 한정)

| 항목 | 선택 | 메모 |
| --- | --- | --- |
| 프레임워크 | **Next.js 14+ (App Router)** | SSG로 빌드. 백엔드 미연결. |
| UI | Tailwind CSS + shadcn/ui | 빠른 프로토타이핑 |
| 지도 | **카카오맵 JavaScript SDK** | 한국 좌표·POI에 최적. API 키만 발급. |
| 캘린더 | `FullCalendar` 또는 자체 그리드 | 1차는 자체 단순 그리드도 OK |
| 데이터 | `/public/data/*.json` fetch | DB 없음 |
| 패키지 매니저 | pnpm 또는 npm | 본인 취향 |
| 빌드 산출물 | `next build && next export` (또는 SSG output) → `out/` 정적 파일 | Nginx로 서빙 |

> **하지 말 것**: 이 단계에서 인증·결제·DB·tRPC 같은 거 절대 끌어들이지 마라. 1차 검증의 목적은 화면이지 아키텍처가 아니다.

## 5. 배포 (오라클 VM + server.ps1)

### 5-1. 사전 준비 (1회성)
- [ ] 오라클 VM에 Node.js 20+, pnpm, Nginx 설치.
- [ ] Nginx에 정적 호스팅 블록 1개 (`root /var/www/vibe;` + `try_files $uri /index.html;`).
- [ ] 방화벽: 오라클 Security List + ufw에 80/tcp 열기 (오라클 자주 빠뜨림).
- [ ] 본인 SSH 키로 비밀번호 없이 접속 확인.

### 5-2. `server.ps1` 표준 흐름
```
# 의도: 로컬 빌드 산출물을 오라클 VM으로 동기화.
1. pnpm install
2. pnpm build  (SSG 출력 -> ./out)
3. rsync -avz --delete ./out/  user@168.x.x.x:/var/www/vibe/
4. ssh user@168.x.x.x "sudo nginx -t && sudo systemctl reload nginx"
5. curl -I http://168.x.x.x/ 로 200 확인
```
- 멱등하게(여러 번 돌려도 같은 결과). 실패 시 빠르게 빠져나가게 (`-ErrorAction Stop`).
- `.env.local` 같은 비밀 값은 Git/스크립트에 절대 안 들어가게 — 이번엔 더미라 비밀 없음.

### 5-3. URL 정책
- 1차: `http://168.x.x.x/` (HTTP, IP).
- **사용자에게 공유 시 주의**: 모바일 사파리/크롬은 HTTP에서 일부 기능(클립보드, 위치, 카메라) 차단. 1차 데모는 카카오맵 표시 정도라 영향 작음.
- 도메인 + HTTPS는 Phase 3에서. 그때 Caddy 또는 Nginx + Let's Encrypt 자동갱신.

| Day 5 | 대시보드 **지도 뷰** (카카오맵 + 핀) | 화면 ②(2/3) |

## 7. 검증 인터뷰 스크립트 (Day 10)

> 5~10명에게 라이브 URL 공유 후 **30분 내** 답을 받는다. 질문 자체보다 "사용자가 어디서 멈추고 어디서 클릭하는지" 관찰하는 것이 핵심.

1. (랜딩 보여주며) **"이 사이트 처음 봤을 때 뭐 하는 곳 같아?"** — 5초 안에 답이 나오는지.
2. **"가장 먼저 누르고 싶은 버튼이 뭐야?"** — D-day vs 대시보드 CTA의 우열.
3. (대시보드 보여주며) **"이 화면에서 가장 먼저 보는 곳이 어디야?"** — 지도/캘린더/리스트 중 자연 선호.
4. **"자체 모임이랑 외부 모임 구분이 보여? 어떤 게 자체야?"** — 색·라벨이 충분한지.
5. **"이 사이트가 한국에 없던 거다 싶어, 아니면 그냥 그래?"** — 어그리게이션의 차별성 체감.
6. **"네가 친구한테 이 링크 공유한다고 가정하면 뭐라고 보낼 거야?"** — 1줄 캐치프레이즈 검증.
7. **"이거 보고 다음 강의 신청할래?"** — 결제 의향 (실 결제는 카톡 오픈채팅으로).

## 8. 검토받을 항목 (사용자 → Claude / 다음 단계 진입 전)

- [ ] 화면 4개 범위가 충분한가, 너무 좁은가?
- [ ] 더미 JSON 스키마가 Phase 2 DB 스키마로 그대로 가도 OK인가?
- [ ] 카카오맵 사용 OK? (네이버 지도로 변경 원하면 알려달라 — SDK 쪽 차이 있음)
- [ ] 외부 모임 시드 12~15건의 출처 표기가 약관·저작권상 OK한 수준인지 본인이 확인.
- [ ] 신청 CTA의 더미 모달 카피("준비 중 — 카톡 오픈채팅으로 문의") OK?
- [ ] `server.ps1`의 흐름(5-2)이 본인이 가진 기존 스크립트와 충돌 없는지.
- [ ] 1차 IP URL 그대로 외부 공유에 OK한가, 아니면 도메인을 1차에 붙일 건가?

## 9. 다음 단계 (Phase 2 진입 조건)

이 1차 데모를 띄우고 **5~10명 인터뷰 결과가 다음 중 하나 이상** 만족하면 백엔드 연결로 진입:
- 인터뷰 응답자 60% 이상이 "이 사이트 한국에 없던 거다"라고 답함.
- "다음 강의 신청 의향 있다"가 30% 이상.
- 사이트 공유 1주일 안에 카톡 오픈채팅 신규 가입 5명 이상.

만족 못하면 **Phase 0(소모임 앱)으로 돌아가** 가격·페르소나·문구를 다듬고 1~2회 강의를 더 굴린 뒤 데모 v2를 만든다. **사이트 짓는 노력보다 강의 자체의 매력 검증이 우선이다.**

## 10. 한 줄 결론

> **1차 데모의 목적은 "동작하는 사이트"가 아니라 "보여주면 사람들이 놀라는 화면"이다.** 어그리게이션 대시보드가 그 역할을 못 하면, 백엔드를 아무리 잘 붙여도 트래픽은 오지 않는다. 5~10일 안에 만들고, 5~10명에게 보여주고, 답을 들고 돌아오자.
