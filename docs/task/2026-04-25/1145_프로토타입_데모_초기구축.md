## 작업 배경

소모임 앱의 기존 바이브코딩 모임 운영 방식(계좌 송금, 회당 ₩22~25k)을 본 뒤,
"사이트 짓기 전 수요 검증" → "프론트 + 더미 JSON으로 화면만 검증" 방향으로 결정.
사용자 추가 요청으로 1차 데모 범위를 화면 4개 → **10개 라우트(어그리게이션 대시보드 + 학습 라이브러리 + 커뮤니티 + 마이 + 어드민)** 로 확장.

## 왜 이 방법을 골랐나 — 주요 결정 4가지

### 1. Next.js + `output: 'export'` (정적 빌드)
- **대안**: Vite + React Router, 또는 동적 SSR.
- **선택 이유**: Phase 1은 백엔드 없음 + 오라클 VM에 Nginx 정적 호스팅이 운영 부담 가장 낮음.
  Next.js의 SSG는 동적 라우트(`/events/[id]`, `/library/[id]`)도 `generateStaticParams` 로 빌드 타임 prerender.
- **Phase 2 백엔드 연결 시 마이그레이션 부담**: 같은 코드베이스에서 `output: 'export'` 만 떼고 API 라우트 추가하면 됨. 페이지 컴포넌트는 그대로.

### 2. 카카오맵 SDK 미사용 → SVG 자체 구현
- **대안**: 카카오/네이버 지도 API.
- **선택 이유**: 1차는 API 키 발급 + 도메인 등록 절차가 데모 출시를 지연시킴. 데모 목적은 "어그리게이션 차별성 체감"이지 정확한 지도가 아님.
- **구현**: 한국 위경도 박스(33~38.7, 124.5~130.5)를 SVG 좌표로 정규화. 자체(`internal`) 강의는 brand-600 + ★ 표시, 외부(`external`)는 slate-400. 핀 호버 시 우측 패널에 카드 표시.
- **Phase 2에서 카카오맵 교체**: `MapView` 컴포넌트만 갈아끼우면 됨. 데이터 모델(lat/lng)은 동일.

### 3. 더미 JSON을 `import` 정적 번들 (vs `fetch /public/data/*.json`)
- **대안**: 클라이언트에서 fetch.
- **선택 이유**: 빌드 타임 import는 SSG 산출물에 데이터가 인라인 → 추가 네트워크 요청 0, 첫 페인트 빠름.
- **타입 안전성 trade-off**: 일단 `as any` 캐스팅으로 통과시켰음. Phase 2에서 zod 스키마 + 타입 추론으로 강화 예정.

### 4. 커뮤니티의 4-탭 구조 (쇼케이스 / 게시판 / Q&A / 회원)
- **대안**: 각각을 최상위 라우트(`/showcase`, `/board`, `/qa`, `/members`)로 분리.
- **선택 이유**: 헤더 nav 항목이 많아지면 모임 정체성이 희석됨. "커뮤니티"라는 단일 입구로 모은 뒤 탭으로 전환하는 게 IA 측면에서 더 정돈되고 향후 추가 탭(예: 채팅·이벤트댓글) 흡수도 쉬움.
- **구현**: `app/community/layout.tsx` 에서 탭 nav를 공유, 각 탭은 자식 라우트.
